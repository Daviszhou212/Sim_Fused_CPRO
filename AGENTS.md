## 目录边界与主线口径

- 一般项目实现、入口配置、测试与实验分析默认优先围绕 `MIMO/` 与 `CLQR/` 展开；除非用户明确另行指定，不要把其他目录当作默认修改目标。
- 查看、解释、复现、比较或新增 **SLDAC 算法** 时，`SLDAC_code/` 是主线和权威基准；不得把 `SLDAC_code/` 当作普通旧参考代码跳过。
- `PRCRL_code/`、`场景参考代码/`、`Backup/` 均视为备份或历史旧参考代码，不属于当前主线实现。
- `SLDAC_code/` 作为 SLDAC 算法主线基准时，默认仍按只读处理：不得擅自修改、重构、同步新功能、迁移配置，或把其中实现复制回其他目录覆盖当前代码。
- 如任务需要修改 `SLDAC_code/` 或上述旧参考目录，必须先取得用户明确确认；如只做对照，应明确说明来源和对照结论。

## SLDAC 算法主线与新增实现兼容基准

- 所有新增或重构的 SLDAC 相关代码，默认以 `SLDAC_code/` 的主线实现作为行为基准；读取 `SLDAC_code/` 时仍按只读规则处理，不得回写。
- 实现前必须只读核对 `SLDAC_code/` 中的关键语义，包括随机流、seed 初始化、环境随机数推进顺序、动作采样、动作有界映射/裁剪、`log_std` 初始化与更新、buffer 存取顺序、cost 向量构造等。
- 特别是 MIMO SLDAC 变体，动作均值/采样、有界处理、非法或非正动作处理应遵照 `SLDAC_code/MIMO1` 的口径；不得在未说明的情况下改成另一套动作分布、裁剪规则或随机流。
- 如新场景或新算法扩展无法完全复刻 `SLDAC_code/`，必须显式说明哪些行为继承主线基准、哪些属于有意偏离，并在配置、注释或测试中留下可追溯记录。

## 强化学习问题核心口径

- 本项目面向的是 `infinite-horizon average reward / average cost` 的 CMDP 场景，不是 discounted MDP / CMDP。
- 除非用户明确要求，否则不得将本项目的问题定义、公式解释、训练逻辑或实验结论按 discounted 口径理解或表述。
- 代码中出现的 `gamma_pow_reward`、`gamma_pow_cost` 等字段，默认应理解为算法内部更新权重相关超参数，不应直接解释为 discounted return 的折扣因子。

- 必须严格区分**环境接口返回的 `reward`**与**算法内部优化的 objective**。
- 在当前主线实现中，`SLDAC` 与 `Fused-CPRO` 都会先将环境 `reward` 转成第 0 维目标成本，即 `objective_cost = -reward`；其余维度表示约束残差。
- 因此，代码中的 `objective_avg`、`func_value[0]`，以及历史命名的 `reward` 曲线、`reward_average_save`、`*_reward.png` 等产物，实际表示的是**待最小化的 objective cost**，解释口径是**越低越好**。
- 阅读代码、撰写注释、分析日志、比较实验结果或修改更新公式时，禁止将“环境原始 reward 的含义”与“算法内部最小化的 objective cost”混为一谈。

## 代码规范补充

- 所有代码文件中，**可调整/可配置参数**应尽量集中在文件最上方定义，避免散落在中下部逻辑中。
- 上述配置项需要添加简洁明确的**中文注释**，说明用途、影响范围与常见取值含义。
- 新增或重构代码时默认遵循该规范；对历史代码按“改到即整理”的原则逐步迁移，不做无关大规模扰动。
- 对于 Python 入口脚本的参数配置，默认应优先保证**直接在 `.py` 中赋值/修改顶部配置即可生效**；除非用户明确要求保留 CLI 覆盖能力，否则不要把 CLI 设计成主要配置入口。

## 实验产物保护（2026-03-24 事故复盘）

- 2026-03-24 曾发生过一次事故：为验证新入口，直接运行脚本覆盖了 `MIMO1` 目录下既有 `Fused_CPRO_*.mat`、`Fused_CPRO_*.png` 等实验产物。后续禁止再次发生同类操作。
- 未经用户**明确确认**，严禁运行任何会写回固定文件名的训练、评估、绘图、导出脚本，尤其是会覆盖 `*.mat`、`*.png`、`*.pdf`、checkpoint 等实验结果的命令。
- 做 smoke test、回归验证、接口检查时，默认优先级必须是：
  1. `py_compile`、静态检查、只读分析。
  2. 函数级最小验证、mock / monkeypatch、内存内检查。
  3. 若必须落盘，只能写入**临时目录、临时文件名、新前缀或显式隔离路径**，不得写回正式实验产物路径。
- 如果脚本默认会写固定产物，且短时间内无法改成临时输出，那么必须先停止执行，并按仓库里的“危险操作确认机制”向用户确认；**不能为了图省事直接运行**。
- 需要生成新的正式实验结果时，必须在执行前明确说明将覆盖的**具体文件路径或文件模式**，并得到用户明确回复 `是 / 确认 / 继续` 后才能执行。
- 对本项目中的 `run_mimo_*`、`CLQR_main.py`、各类 plot/export 脚本，一律默认视为**可能覆盖正式实验产物**；除非已经确认输出到隔离路径，否则不得直接运行。
- 所有 smoke test、回归验证、临时仿真产生的落盘产物，在仿真完成后必须立即移动到对应目录下名为 `Trash` 的文件夹中（例如 `MIMO1/Trash`、`CLQR1/Trash`）；不得把这类临时 `*.mat`、`*.png`、`*.pdf`、checkpoint 长留在正式结果目录里。
